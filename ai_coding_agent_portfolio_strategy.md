# AI Coding Agent 기반 포트폴리오 게시 전략

## 1. 전체 포지셔닝

이 포트폴리오는 전통적인 의미의 “코드를 직접 작성하는 개발자 포트폴리오”로 보이기보다, **AI Coding Agent를 활용해 현업 문제를 작동 가능한 프로토타입으로 구현하고 검증하는 LLM PoC Builder 포트폴리오**로 보여주는 것이 좋다.

핵심 메시지는 다음과 같다.

> 현장의 문제를 구조화하고, AI Coding Agent를 활용해 검증 가능한 LLM 프로토타입으로 구현합니다.

또는 조금 더 솔직하고 직무형으로 쓰면 다음과 같다.

> 고객센터 현장에서 출발해 RAG, Multi-Agent, AI Coding Workflow를 실험해 온 LLM PoC 지향 Builder입니다. 코드를 직접 작성하는 역량보다, 문제를 구조화하고 AI Coding Agent를 활용해 작동하는 프로토타입으로 구현·검증하는 역량을 강점으로 갖고 있습니다.

## 2. 포트폴리오에서 숨기지 말아야 할 것

AI Coding Agent를 활용해 구현했다는 사실은 숨기지 않는 것이 좋다. 다만 “코드를 한 줄도 못 짠다”처럼 표현하면 역할이 너무 약하게 보일 수 있다.

### 피해야 할 표현

> 저는 코드를 작성할 줄 모릅니다.

### 권장 표현

> 코드는 AI Coding Agent를 활용해 구현했으며, 저는 문제 정의, 기능 요구사항 정리, 데이터 흐름 설계, 아키텍처 방향 설정, 구현 결과 검증을 담당했습니다.

또는 다음처럼 더 구체적으로 표현할 수 있다.

> 저는 요구사항 정의, 화면 흐름 설계, 데이터 흐름 설계, Agent에게 전달할 구현 지시 작성, 구현 결과 테스트, 오류 재현 및 수정 방향 제시, 문서화와 개선 반복을 담당했습니다.

이렇게 쓰면 AI를 활용했다는 점이 약점이 아니라, **현대적인 AI-assisted development 역량**으로 보인다.

## 3. 포트폴리오 메인 카피 추천

### 짧은 버전

> 현업 문제를 AI Coding Agent로 제품화하는 LLM PoC Builder입니다.

### 조금 더 설명적인 버전

> 고객센터 업무 자동화에서 시작해 RAG, Multi-Agent SaaS, Obsidian 기반 AI Wiki까지 실험하고 있습니다. 저는 문제 정의, 데이터 흐름 설계, Agent 구현 지시, 결과 검증을 중심으로 AI-assisted development를 수행합니다.

### 가장 추천하는 버전

> 현장의 문제를 구조화하고, AI Coding Agent를 활용해 검증 가능한 LLM 프로토타입으로 구현합니다.

## 4. 프로젝트 노출 순서

### 1순위. 고객센터 KPI 모니터링·업무 자동화 웹 앱

이 프로젝트를 가장 위에 두는 것이 좋다.

이유는 다음과 같다.

- 실제 업무 문제에서 출발했다.
- 실제 사용자가 있었다.
- 실제 조직 KPI 개선과 연결된다.
- 왜 개발과 AI에 관심을 갖게 되었는지 설명할 수 있다.
- AI Coding Agent 활용의 출발점이다.

이 프로젝트는 포트폴리오 전체의 스토리 중심축이다.

### 2순위. 하이브리드 RAG 카드 추천 시스템

LLM PoC 직무와 가장 직접적으로 연결되는 프로젝트다.

강조할 포인트는 다음과 같다.

- RAG 구조 설계
- 검색 품질 개선
- 평가 모드 설계
- 데이터 정리와 검색 구조의 중요성
- 모델 성능 차이를 검색 로직으로 일부 상쇄할 수 있다는 실험 경험

### 3순위. CrewAI 기반 Multi-Agent SaaS Wrapper

가장 야심찬 프로젝트다. 다만 “상용 SaaS를 완성했다”처럼 보이면 과장으로 읽힐 수 있으므로 **SaaS형 프로토타입**이라고 표현하는 것이 안전하다.

강조할 포인트는 다음과 같다.

- Agent Workflow를 일반 사용자가 구성할 수 있게 만든 Builder 구조
- Agent, Task, Crew, Flow 등 추상적인 개념을 사용자 흐름으로 바꾼 경험
- 편집 상태와 실행 상태를 분리한 Published Runtime Snapshot 구조

### 4순위. Obsidian AI Wiki 프로젝트

현재 진행 중인 프로젝트로 따로 보여주는 것이 좋다.

카테고리는 다음처럼 잡는 것이 적절하다.

> 현재 진행 중인 실험: AI Coding Agent의 컨텍스트 관리와 문서 기반 실행 구조

이 프로젝트는 완성도보다 문제의식과 방향성이 중요하다.

## 5. 프로젝트 카드 공통 구조

각 프로젝트는 아래 구조로 통일하는 것이 좋다.

```md
## 프로젝트명

### 한 줄 설명

### 문제
왜 만들었는가?

### 내 역할
AI Coding Agent를 어떻게 활용했고, 나는 무엇을 판단했는가?

### 핵심 구현
너무 기술을 많이 나열하지 말고 3개 정도만 정리한다.

### 성과
수치, 검증 결과, 배운 점을 정리한다.

### 한계와 개선
솔직하게 1~2줄 작성한다.
```

특히 **한계와 개선**을 넣는 것이 중요하다. AI Coding Agent 기반 구현자는 과장된 성과보다, 한계를 인식하고 검증하며 개선한 태도가 더 신뢰를 준다.

# 프로젝트별 게시 문구 예시

## 1. 고객센터 KPI 모니터링·업무 자동화 웹 앱

### 한 줄 소개

상담원 개인 PC에서 팀 전체 KPI와 인입 현황을 실시간으로 확인할 수 있도록 만든 내부 업무 개선 웹 앱입니다.

### 문제

기존에는 고객 인입 현황을 사무실 천장에 설치된 TV 한 대로만 확인해야 했습니다. 상담원이 화면을 의식적으로 보지 않으면 대기 상황을 체감하기 어려웠고, 일부 인원에게 업무 부담이 집중되는 문제가 반복되었습니다.

### 내 역할

문제 정의, 화면 흐름 설계, 데이터 흐름 설계, AI Coding Agent 기반 웹 프로토타입 구현, 배포, 사용자 피드백 반영을 담당했습니다.

### 핵심 구현

- 상담원 개인 PC에서 확인 가능한 실시간 KPI 대시보드
- 이메일 자동화 도구를 통한 응대 형식 표준화
- 생산성, 평균 통화 시간, 휴식 사용 지표 기반의 팀 운영 보조

### 성과

대시보드 데이터는 저성과자 교육과 팀 운영 판단에 활용되었고, 담당 팀은 월별 KPI에서 생산·영업·품질 지표 기준 5개 팀 중 1위를 달성했습니다. 기능 확장 과정에서 회귀 버그와 문서화 부족을 겪으며, 이후 프로젝트에서 테스트와 유지보수 구조를 더 중요하게 보게 되었습니다.

### 한계와 개선

초기에는 빠른 구현에 집중해 문서화와 테스트가 부족했습니다. 이후 프로젝트에서는 기능을 추가하기 전에 구조와 데이터 흐름을 먼저 정리하는 방식으로 개선하고 있습니다.

## 2. 하이브리드 RAG 카드 추천 시스템

### 한 줄 소개

카드 혜택 데이터를 검색하고 사용자 조건에 맞는 추천 근거를 생성하는 RAG 기반 추천 프로토타입입니다.

### 문제

카드 혜택 데이터는 표현 방식이 제각각이라, 단순 검색만으로는 사용자의 조건에 맞는 혜택을 안정적으로 찾기 어려웠습니다.

### 내 역할

RAG 구조 설계, 검색 전략 고도화, LangChain 체인 구성, 평가 모드 설계를 담당했습니다. 코드는 AI Coding Agent의 도움을 받아 구현했고, 저는 검색 흐름과 평가 기준을 설계했습니다.

### 핵심 구현

- 혜택 단위 chunking과 메타데이터 기반 검색
- 키워드 검색과 벡터 검색을 결합한 하이브리드 검색
- Rerank, Hard Filter, Multi Query, RRF 기반 성능 비교

### 성과

기본 검색 모델보다 커스텀 RAG 구조가 더 높은 승률을 보이는 결과를 확인했습니다. 특히 검색 로직을 개선하면 모델 성능 차이를 일부 상쇄할 수 있다는 점을 실험으로 검증했습니다.

### 한계와 개선

성능 비교 결과는 내부 평가 기준에 기반한 것이므로, 포트폴리오에는 평가 방식과 샘플 질문을 함께 공개하는 것이 좋습니다. 가능하다면 승률 표나 비교 그래프를 함께 제시하면 신뢰도가 높아집니다.

## 3. CrewAI 기반 Multi-Agent SaaS Wrapper

### 한 줄 소개

사용자가 Agent Workflow를 구성하고 실행할 수 있도록 만든 SaaS형 Multi-Agent 오케스트레이션 프로토타입입니다.

### 문제

CrewAI 같은 Agent 프레임워크는 강력하지만, 일반 사용자가 직접 Agent, Task, Flow를 구성하고 실행하기에는 추상도가 높고 사용 흐름이 복잡했습니다.

### 내 역할

LLM 서비스 기획, Workflow 구성 요소 설계, Builder UX 흐름 설계, AI Coding Agent 기반 주요 프론트엔드 기능 및 백엔드 도메인 로직 구현을 담당했습니다.

### 핵심 구현

- Agent, Task 등 Workflow 구성 요소 CRUD
- React Flow 기반 Crew/Flow Builder
- 편집 상태와 실행 상태를 분리한 Published Runtime Snapshot 구조

### 성과

사용자가 직접 Agent Workflow를 구성하고 실행할 수 있는 SaaS형 오케스트레이션 플랫폼 구조를 프로토타입으로 구현했습니다. 특히 Builder의 편집 상태와 실행 상태를 분리해, 실행 시점에는 발행된 snapshot 기준으로 동작하도록 설계했습니다.

### 한계와 개선

현재 단계에서는 상용 SaaS가 아니라 프로토타입 수준으로 표현하는 것이 좋습니다. 포트폴리오에서는 “완성된 서비스”보다 “복잡한 Agent Workflow를 사용자 친화적인 Builder 구조로 바꾸려 한 시도”를 강조하는 것이 적절합니다.

## 4. Obsidian AI Wiki 프로젝트

### 한 줄 소개

AI Coding Agent가 참고할 문서, 실행 흐름, 결과물을 관리하기 위한 Obsidian 기반 Agentic Engineering 실험입니다.

### 문제

AI Coding Agent를 활용한 개발에서는 기능이 커질수록 오래된 맥락, 불필요한 파일 참조, 문서화 부족으로 잘못된 구현이 반복될 수 있습니다.

### 내 역할

문제 정의, 문서 구조 설계, Agent 실행 흐름 설계, 입출력 계약 설계, Codex CLI 기반 실행 구조 실험을 진행하고 있습니다.

### 핵심 구현 또는 설계

- Wiki 기반 프로젝트 문서화 구조
- Agent가 참고할 문서 범위와 실행 흐름 정의
- Runtime, Middleware, 입출력 계약 설계
- 코드 생성 결과의 추적과 재사용을 위한 문서 관리 방식

### 현재 상태

아직 진행 중인 프로젝트이며, 목표는 AI Coding Agent를 더 안정적으로 활용하기 위한 개발 문서화·실행 관리 구조를 만드는 것입니다.

### 한계와 개선

완성 프로젝트로 보이기보다 현재 진행 중인 연구·실험 프로젝트로 보여주는 것이 좋습니다. 이 프로젝트는 결과물보다 문제의식, 구조화 방식, 문서화 철학을 보여주는 용도로 활용하는 것이 적절합니다.

- 데이터 흐름도

- Agent Workflow 구조도

- Before / After 비교
- 테스트 결과 표
- 회고 문서
- 구현 과정에서 작성한 프롬프트 또는 작업 지시 예시

# 전체 게시 전략 요약

## 1단계. 솔직하게 포지셔닝한다

전통적인 코더처럼 보이려고 하기보다, AI Coding Agent를 활용해 현업 문제를 구현 가능한 프로토타입으로 바꾸는 사람으로 정의한다.

## 2단계. 프로젝트는 문제 해결 흐름 중심으로 작성한다

각 프로젝트는 다음 흐름을 따른다.

> 문제 → 접근 → 구현 → 검증 → 한계와 개선

## 3단계. 코드보다 설계 산출물을 함께 보여준다

포트폴리오에서는 코드 자체보다 문제 정의, 데이터 흐름, 검증 방식, 문서화 습관이 드러나야 한다.

## 4단계. GitHub는 코드 실력 증명보다 작업 방식 증명으로 활용한다

README에는 AI Coding Agent 기반 구현 사실과 본인의 역할을 명확히 적는다. 이것은 약점이 아니라 작업 방식의 투명성으로 보여야 한다.
